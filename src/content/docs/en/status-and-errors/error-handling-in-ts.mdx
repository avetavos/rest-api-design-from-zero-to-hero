Scattering error responses through every handler guarantees inconsistency. Instead, throw typed errors in your handlers and translate them to `problem+json` in one place.

## A typed error + a central handler

Define an error type that carries the status and problem fields, then a single handler that converts it. Anything unexpected becomes a `500` — and you never leak a stack trace to the client.

export const honoCode = `class ApiError extends Error {
  constructor(status, title, detail) {
    super(title);
    this.status = status;
    this.title = title;
    this.detail = detail;
  }
}

app.get('/articles/:id', (c) => {
  if (c.req.param('id') === '0') {
    throw new ApiError(404, 'Article not found', 'No article with id 0');
  }
  return c.json({ id: c.req.param('id'), title: 'Hello' });
});

// Central error handler: everything funnels through here.
app.onError((err, c) => {
  if (err instanceof ApiError) {
    return c.json(
      { type: 'about:blank', title: err.title, status: err.status, detail: err.detail },
      err.status,
      { 'Content-Type': 'application/problem+json' },
    );
  }
  // Unexpected: log server-side, return a safe 500.
  console.error(err);
  return c.json(
    { type: 'about:blank', title: 'Internal Server Error', status: 500 },
    500,
    { 'Content-Type': 'application/problem+json' },
  );
});`;

## Why centralize

- **Consistency** — every error leaves through the same shape.
- **Safety** — unexpected errors become a generic `500`; internals never leak.
- **Less code** — handlers throw a line; they do not each build a response.

<Callout title="Throw typed errors, translate once">
Let handlers express failure by throwing a typed error (`throw new ApiError(404, ...)`), and put all the formatting in one `onError` hook. You get uniform `problem+json` everywhere, a guaranteed safe fallback for the unexpected, and handlers that stay focused on the happy path.
</Callout>
