## The checklist

**Resources & URIs** — model nouns, not actions; plural collections (`/articles`); lowercase-hyphenated paths; IDs in the path; nest only for ownership, otherwise link. See [Resource & URI Design](/rest-api/en/resource-and-uri-design/).

**Methods & idempotency** — map CRUD to GET/POST/PUT/PATCH/DELETE; never mutate on GET; make writes idempotent where you can; protect non-idempotent creates with idempotency keys. See [Methods, CRUD & Idempotency](/rest-api/en/methods-crud-idempotency/).

**Status & errors** — pick the precise status code; never return `200` for a failure; use one consistent `problem+json` error shape; return all validation errors at once. See [Status Codes & Errors](/rest-api/en/status-and-errors/).

**Querying collections** — always paginate (prefer cursor for large/changing data); allowlist filter/sort fields; offer sparse fieldsets and a consistent envelope. See [Querying Collections](/rest-api/en/collections-querying/).

**Versioning & caching** — favor additive, non-breaking changes; version coarsely when you must; set `Cache-Control` and `ETag`s; use conditional requests for concurrency. See [Versioning & Caching](/rest-api/en/versioning-caching/).

**Security** — HTTPS only; never put secrets in URLs; distinguish `401`/`403`; verify tokens (do not just decode); tight CORS allowlist; rate-limit and validate all input. See [Security](/rest-api/en/security/).

**Ship it** — describe the API with OpenAPI; test unit/integration/contract layers; keep handlers thin and services HTTP-free. See [OpenAPI](/rest-api/en/docs-testing-building/openapi/).

<Callout title="Consistency beats cleverness">
The best APIs are not the cleverest — they are the most predictable. Apply these conventions uniformly and a developer who has used one of your endpoints can guess the rest. That predictability, more than any single feature, is what makes an API a pleasure to build against.
</Callout>
