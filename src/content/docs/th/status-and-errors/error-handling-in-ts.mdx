การกระจาย error response ไปทั่วทุก handler รับประกันความไม่สม่ำเสมอ แทนที่จะทำเช่นนั้น ให้ throw typed error ใน handler ของคุณ แล้วแปลงเป็น `problem+json` ที่จุดเดียว

นิยาม error type ที่พก status และ field ของ problem ไว้ จากนั้นมี handler เดียวที่แปลงมัน อะไรที่ไม่คาดคิดจะกลายเป็น `500` — และคุณจะไม่รั่ว stack trace ออกไปยัง client เลย

export const honoCode = `class ApiError extends Error {
  constructor(status, title, detail) {
    super(title);
    this.status = status;
    this.title = title;
    this.detail = detail;
  }
}

app.get('/articles/:id', (c) => {
  if (c.req.param('id') === '0') {
    throw new ApiError(404, 'Article not found', 'No article with id 0');
  }
  return c.json({ id: c.req.param('id'), title: 'Hello' });
});

// Central error handler: everything funnels through here.
app.onError((err, c) => {
  if (err instanceof ApiError) {
    return c.json(
      { type: 'about:blank', title: err.title, status: err.status, detail: err.detail },
      err.status,
      { 'Content-Type': 'application/problem+json' },
    );
  }
  // Unexpected: log server-side, return a safe 500.
  console.error(err);
  return c.json(
    { type: 'about:blank', title: 'Internal Server Error', status: 500 },
    500,
    { 'Content-Type': 'application/problem+json' },
  );
});`;

## ทำไมต้องรวมศูนย์

- **ความสม่ำเสมอ** — ทุก error ออกไปด้วยรูปทรงเดียวกัน
- **ความปลอดภัย** — error ที่ไม่คาดคิดกลายเป็น `500` แบบทั่วไป; ส่วนภายในไม่รั่วไหล
- **โค้ดน้อยลง** — handler แค่ throw หนึ่งบรรทัด; ไม่ต้องสร้าง response เองทุกตัว

<Callout title="โยน typed error แล้วแปลงที่เดียว">
ให้ handler แสดงความล้มเหลวด้วยการ throw typed error (`throw new ApiError(404, ...)`) และนำการจัดรูปแบบทั้งหมดไปไว้ใน hook `onError` เดียว คุณจะได้ `problem+json` ที่เหมือนกันทุกที่ มี fallback ที่ปลอดภัยรับประกันสำหรับสิ่งที่ไม่คาดคิด และ handler ที่จดจ่ออยู่กับ happy path
</Callout>
