วิธีที่ API รายงานผลลัพธ์เป็นส่วนหนึ่งของ contract ของมัน สองสิ่งที่ทำให้ contract นั้นน่าเชื่อถือคือ **status code** (ส่วนที่ client ใช้แตกกิ่งเงื่อนไข) และ **error body ที่สม่ำเสมอ** (ส่วนที่คนและ code อ่านเพื่อกู้สถานการณ์)

## สำเร็จ, client error หรือ server fault?

กฎที่สำคัญที่สุดเพียงข้อเดียว: อย่าห่อความล้มเหลวไว้ใน `200` เด็ดขาด การตอบกลับที่บอกว่า `200 OK` แต่กลับแนบ `{ "error": "..." }` มาด้วยนั้นทำลายทุก client, cache และเครื่องมือ monitoring ที่เชื่อถือ status line

## โมดูลนี้ครอบคลุมอะไรบ้าง

- **การเลือก status code** — code ที่พบบ่อยและวิธีเลือกในกรณีที่ใกล้เคียงกัน
- **รูปแบบของ error (problem+json)** — รูปทรงเดียวที่สม่ำเสมอสำหรับทุก error
- **Validation error** — รายงาน input ที่ผิดทีละ field
- **การจัดการ error ใน TypeScript** — แปลง error ที่ถูก throw ให้กลายเป็น response ที่สะอาด

<Callout title="status code คือคำตอบของ API">
client จะแตกกิ่งเงื่อนไขจาก status code ก่อน แล้วจึงอ่าน body ทีหลัง ถ้าเลือก code ถูก — `2xx`/`4xx`/`5xx` — แม้แต่ client ที่ไม่สนใจ body ของคุณก็จะทำงานได้ถูกต้อง แต่ถ้าเลือกผิด รายละเอียดใน body มากแค่ไหนก็ช่วยพวกเขาไม่ได้
</Callout>
