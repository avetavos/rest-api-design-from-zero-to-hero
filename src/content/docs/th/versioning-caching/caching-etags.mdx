HTTP มีโมเดล caching ที่ทรงพลังติดตัวมาอยู่แล้ว การใช้มันให้ดีหมายความว่า client (หรือ CDN) สามารถหลีกเลี่ยงการดาวน์โหลดข้อมูลที่ไม่ได้เปลี่ยนแปลงซ้ำได้ — ซึ่งมักจะเป็นชัยชนะด้านประสิทธิภาพชิ้นเดียวที่ใหญ่ที่สุดที่ API จะได้รับ

## Cache-Control: นี่จะ fresh นานแค่ไหน?

`Cache-Control` บอก cache ว่า response สามารถนำกลับมาใช้ซ้ำได้หรือไม่และนานเท่าใด:

```http
Cache-Control: public, max-age=60        # any cache may reuse for 60s
Cache-Control: private, max-age=0        # only the client, must revalidate
Cache-Control: no-store                  # never cache (sensitive data)
```

ในขณะที่ response ยัง *fresh* อยู่ client จะนำมันกลับมาใช้ซ้ำโดยไม่ต้องส่ง request เลย เมื่อมันกลายเป็น stale แล้ว มันจึงจะ revalidate

## ETag + If-None-Match: มันเปลี่ยนหรือเปล่า?

`ETag` คือ validator — ลายนิ้วมือของ representation ปัจจุบัน ใน request ถัดไป client จะส่งมันกลับมาใน `If-None-Match`; หากมันยังคงตรงกัน server จะคืน `304 Not Modified` โดยไม่มี body:

`Last-Modified` + `If-Modified-Since` คือสิ่งเทียบเท่าที่อิงตาม timestamp ETag แม่นยำกว่า (มันจับการเปลี่ยนแปลงใด ๆ ได้ ไม่ใช่แค่การเปลี่ยนแปลงเวลาในระดับวินาที)

## การคำนวณ ETag

export const etagCode = `function etagFor(resource) {
  const json = JSON.stringify(resource);
  let hash = 0;
  for (let i = 0; i < json.length; i++) hash = (hash * 31 + json.charCodeAt(i)) | 0;
  return '"' + (hash >>> 0).toString(16) + '"';
}

const article = { id: 42, title: 'Hello', version: 7 };
const tag = etagFor(article);
console.log('ETag:', tag);

// Conditional request: client sends back the tag it has.
function handle(ifNoneMatch) {
  return ifNoneMatch === tag ? 304 : 200;
}
console.log('with stale tag:', handle('"old"'));   // 200
console.log('with current tag:', handle(tag));      // 304`;

| Cache Mechanism | ทำงานอย่างไร | เหมาะกับ |
|---|---|---|
| `Cache-Control: max-age=3600` | browser cache นาน 1 ชั่วโมง | static resource, slow-changing data |
| ETag + `If-None-Match` | server return 304 ถ้า resource ไม่เปลี่ยน | resource ที่เปลี่ยนบ้าง |
| `Last-Modified` + `If-Modified-Since` | ใช้ timestamp แทน hash | resource ที่รู้ modified time ชัดเจน |
| `Cache-Control: no-store` | ไม่ cache เลย | sensitive data เช่น payment, user profile |

## Common Mistakes

**Cache Sensitive Data โดยไม่ตั้งใจ**
อาการ:
- `Cache-Control: max-age=3600` บน `/users/\{id\}/profile`
- browser หรือ CDN cache profile ของ user A ให้ user B เห็น
- ตั้ง `Cache-Control: no-store, private` สำหรับ authenticated resource

**ETag ที่ไม่ Stable**
อาการ:
- ETag generate จาก timestamp — server ที่ต่างกัน generate ETag ต่างกันสำหรับ content เดียว
- client ส่ง `If-None-Match` แต่ server return 200 ทุกครั้งแทน 304
- generate ETag จาก hash ของ content แทน timestamp

> 💡 Real World Example
>
> **GitHub API:**
> - ETag บน resource ทุกตัว — conditional GET ลด bandwidth 90%+
> - rate limit ไม่นับ request ที่ return 304
>
> **AWS S3:**
> - ETag คือ MD5 hash ของ object content
> - conditional GET ผ่าน `If-None-Match` ทำให้ sync ไฟล์จำนวนมากได้อย่างมีประสิทธิภาพ

<Callout title="ตั้ง Cache-Control บนทุก response">
แม้แต่ `Cache-Control: no-store` ก็เป็นการตัดสินใจที่ควรระบุออกมาอย่างชัดเจน — มันบอก cache ว่าควรทำอะไรแทนที่จะปล่อยให้คลุมเครือ เพิ่ม `ETag` ให้กับ representation ที่คำนวณหรือถ่ายโอนแล้วมีต้นทุนสูง, และ conditional requests จะเปลี่ยนการตรวจสอบแบบ "มันยังเป็นปัจจุบันอยู่ไหม?" ส่วนใหญ่ให้กลายเป็น `304` ที่ราคาถูก
</Callout>
