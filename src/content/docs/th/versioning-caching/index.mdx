มีสองแรงที่กำหนดรูปร่างของ API ที่มีอายุยืนยาว: มันต้อง **เปลี่ยนแปลง** ได้โดยไม่ทำให้ client ที่ใช้งานอยู่แล้วพัง, และมันควรจะ **เร็ว** โดยในอุดมคติคือไม่ทำงานซ้ำในสิ่งที่เคยทำไปแล้ว Versioning จัดการเรื่องแรก; HTTP caching จัดการเรื่องที่สอง

ทั้งสองเรื่องนี้เกี่ยวข้องกัน: caching ขึ้นอยู่กับ representation ที่เสถียร, และ versioning เป็นตัวกำหนดว่า representation เหล่านั้นจะมีหน้าตาเป็นอย่างไรเมื่อเวลาผ่านไป

## โมดูลนี้ครอบคลุมอะไรบ้าง

- **Versioning strategies** — versioning แบบ URI, header, และ media-type
- **Content negotiation** — การเลือก representation ด้วย `Accept`
- **Caching และ ETags** — `Cache-Control`, validator, และ `304 Not Modified`
- **Conditional requests และ idempotency keys** — การเขียนและการ retry ที่ปลอดภัย

<Callout title="เวอร์ชันที่ถูกที่สุดคือการไม่ต้องมีเวอร์ชันใหม่">
การเปลี่ยนแปลงส่วนใหญ่สามารถทำแบบเพิ่มเติม (additive) ได้ — field ใหม่ที่เป็น optional, endpoint ใหม่ — ซึ่ง client ที่มีอยู่เดิมก็แค่ละเลยมันไป เก็บ API เวอร์ชันใหม่ไว้สำหรับการเปลี่ยนแปลงที่เป็น breaking change จริง ๆ เท่านั้น ทุกเวอร์ชันที่คุณปล่อยออกไปคือเวอร์ชันที่คุณต้องดูแลรักษา ดังนั้นจงตั้งมาตรฐานให้สูงก่อนที่จะสร้างเวอร์ชันใหม่
</Callout>
