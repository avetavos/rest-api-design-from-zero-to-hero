endpoint แบบ collection อย่าง `GET /articles` อาจคืนค่ามาแค่สิบแถวหรือสิบล้านแถวก็ได้ การคืนทั้งหมดออกมาไม่เคยเป็นคำตอบที่ถูกต้อง ความสามารถในการ query สี่อย่างจะเปลี่ยน list ดิบ ๆ ให้กลายเป็นสิ่งที่ client ใช้งานได้อย่างมีประสิทธิภาพ

## คันโยกทั้งสี่

ทั้งสี่อย่างนี้อยู่ใน **query string** เพราะมันอธิบายว่า *คุณต้องการมุมมองแบบไหนของ collection* ไม่ใช่ *ต้องการ resource ตัวไหน* ตัวตน (identity) อยู่ที่ path ส่วน query เป็นตัวกำหนดรูปร่างของผลลัพธ์

## โมดูลนี้ครอบคลุมอะไรบ้าง

- **Pagination** — offset เทียบกับ cursor และ response envelope
- **Filtering & sorting** — การเลือกและจัดลำดับแถว
- **Field selection** — sparse fieldsets และการ expand ความสัมพันธ์
- **Search & envelopes** — full-text search และ wrapper ที่สม่ำเสมอ

<Callout title="ทำ pagination ให้ collection เสมอโดยปริยาย">
อย่าคืน list แบบไม่จำกัดจำนวน ให้ใส่ค่า page size เริ่มต้นที่สมเหตุสมผลแม้ client จะไม่ได้ร้องขออะไรมา และจำกัดค่าสูงสุดไว้ด้วย endpoint ที่คืนทุกอย่างทำงานได้ดีตอนพัฒนา แต่จะล่มในวันที่ตารางข้อมูลโตขึ้น
</Callout>
