สาม method นี้ครอบคลุมส่วน "U" และ "D" ของ CRUD จุดที่ละเอียดอ่อนคือความแตกต่างระหว่าง `PUT` และ `PATCH` — ทำให้ถูกต้องแล้วการอัปเดตจะคาดเดาได้เสมอ

## PUT — การแทนที่ทั้งหมด

`PUT` แทนที่ resource ที่ URI หนึ่งด้วย representation ที่อยู่ใน body ให้ส่ง resource **ทั้งหมด** ไป ฟิลด์ใดก็ตามที่คุณละเว้น ในเชิงความหมายคือกำลังถูกตั้งให้เป็นค่าว่าง เพราะผลลัพธ์ของการ `PUT` body เดิมสองครั้งเหมือนกับการทำครั้งเดียว `PUT` จึงเป็น **idempotent**

```http
PUT /articles/42 HTTP/1.1
Content-Type: application/json

{ "title": "Updated title", "body": "Full new body" }
```

## PATCH — การอัปเดตบางส่วน

`PATCH` ใช้การแก้ไขแบบ **บางส่วน** มีสองรูปแบบที่พบบ่อย:

- **JSON Merge Patch** (`application/merge-patch+json`): ส่งเฉพาะฟิลด์ที่จะเปลี่ยน โดย `null` หมายถึง "ลบฟิลด์นี้ออก"
- **JSON Patch** (`application/json-patch+json`): ส่ง array ของ operation ที่ระบุชัดเจน (`add`, `remove`, `replace`, ...)

โดยทั่วไป `PATCH` ไม่รับประกันว่าจะ idempotent (operation อย่าง "ต่อท้ายลงใน list" ไม่ idempotent) แม้ว่า merge-patch หลายกรณีจะบังเอิญ idempotent ก็ตาม

## DELETE — การลบ

`DELETE` ลบ resource ออก มันเป็น idempotent เพราะการลบ resource ที่ถูกลบไปแล้วก็ยังคงทำให้มันถูกลบอยู่ดี ให้คืน `204 No Content` (หรือ `200 OK` พร้อม body) เมื่อสำเร็จ ส่วน `DELETE` ครั้งที่สองอาจคืน `204` หรือ `404` ขึ้นอยู่กับนโยบายของคุณ — เลือกอย่างใดอย่างหนึ่งและทำให้สม่ำเสมอ

## การอัปเดตใน API ที่กำลังทำงานจริง

export const honoCode = `const store = new Map([[1, { id: 1, title: 'Original', body: 'Hello' }]]);

// PUT — full replacement (idempotent)
app.put('/articles/:id', async (c) => {
  const id = Number(c.req.param('id'));
  const body = await c.req.json();
  const replaced = { id, title: String(body.title ?? ''), body: String(body.body ?? '') };
  store.set(id, replaced);
  return c.json(replaced);
});

// PATCH — merge only provided fields
app.patch('/articles/:id', async (c) => {
  const id = Number(c.req.param('id'));
  const current = store.get(id);
  if (!current) return c.json({ error: 'Not found' }, 404);
  const patch = await c.req.json();
  const updated = { ...current, ...patch, id };
  store.set(id, updated);
  return c.json(updated);
});

app.delete('/articles/:id', (c) => {
  store.delete(Number(c.req.param('id')));
  return c.body(null, 204); // idempotent
});`;

| Method | ส่ง Body? | Idempotent? | ใช้เมื่อ |
|---|---|---|---|
| PUT | ✅ ทั้ง resource | ✅ | แทนที่ resource ทั้งหมด |
| PATCH | ✅ เฉพาะที่แก้ | อาจใช่ | แก้บาง field |
| DELETE | บางครั้ง | ✅ | ลบ resource |

## Common Mistakes

**ใช้ PUT เพื่อ Partial Update**
อาการ:
- `PUT /users/\{id\}` ส่งแค่ `{ "name": "New Name" }` — field อื่น reset เป็น null
- PUT ต้องส่ง resource ทั้งหมด — ขาด field = null/default
- ใช้ PATCH สำหรับ partial update

**DELETE ที่ไม่ Idempotent**
อาการ:
- `DELETE /users/\{id\}` ครั้งแรก return 200, ครั้งที่สอง return 500 (server error)
- ควร return 404 หรือ 204 ในครั้งที่สอง ไม่ใช่ 500
- DELETE ที่ resource ไม่มีแล้วควร return 404 หรือ 204 สม่ำเสมอ

> 💡 Real World Example
>
> **GitHub API:**
> - `PATCH /repos/\{owner\}/\{repo\}` — แก้บาง setting ของ repo
> - `DELETE /repos/\{owner\}/\{repo\}` — ลบ repo (idempotent — 404 ครั้งที่สอง)
>
> **Stripe API:**
> - PATCH สำหรับ update customer info
> - ไม่มี true DELETE สำหรับ charge — ใช้ refund แทน (audit trail)

<Callout title="PUT แทนที่ ส่วน PATCH รวม (merge)">
ถ้า client ส่ง resource ที่สมบูรณ์มาและคาดหวังให้ฟิลด์ที่ละเว้นถูกล้างค่า นั่นคือ `PUT` ถ้า client ส่งเฉพาะสิ่งที่เปลี่ยนและคาดหวังให้สิ่งอื่นทั้งหมดคงไว้ นั่นคือ `PATCH` การเลือก semantics แบบ `PUT` แต่กลับ merge (หรือในทางกลับกัน) คือสาเหตุที่พบบ่อยของบั๊กแบบ "ทำไมฟิลด์ของฉันหายไป?"
</Callout>
