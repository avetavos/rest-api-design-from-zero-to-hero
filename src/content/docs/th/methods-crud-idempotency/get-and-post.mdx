`GET` และ `POST` คือสอง method ที่คุณจะใช้บ่อยที่สุด ทั้งคู่อยู่คนละขั้วของสเปกตรัมเรื่อง safety โดย `GET` ทำได้แค่อ่าน ในขณะที่ `POST` ใช้สร้างและเป็น method ที่มีข้อจำกัดน้อยที่สุดในบรรดาทั้งหมด

## GET — ปลอดภัยและแคชได้

`GET` ดึงตัวแทน (representation) ของ resource มา และต้องไม่เปลี่ยนแปลง state ของ server เด็ดขาด เพราะมันปลอดภัย response จึงสามารถถูกแคชได้และคำขอสามารถ retry ได้อย่างอิสระ `GET` บน collection จะคืนรายการกลับมา ส่วน `GET` บน item เดียวจะคืน item นั้นหรือ `404`

```http
GET /articles/42 HTTP/1.1
Accept: application/json
```

`GET` ไม่ควรพก request body ที่มีความหมายติดไปด้วย และต้องไม่มี side effect — ห้ามทำอย่าง "เพิ่มยอดวิวผ่าน GET" ไม่เช่นนั้น cache และ prefetcher จะสร้างเรื่องเซอร์ไพรส์ให้คุณ

## POST — การสร้างและการกระทำที่ไม่ idempotent

`POST` ส่งข้อมูลไปยัง collection ซึ่งโดยทั่วไปคือการสร้าง resource ใหม่ มันไม่ทั้ง safe และไม่ idempotent การส่ง `POST` เดิมซ้ำสองครั้งมักจะสร้าง resource ขึ้นสองตัว เมื่อสำเร็จ ให้คืน `201 Created` พร้อม header `Location` ที่ชี้ไปยัง resource ใหม่

## ทั้งสองอย่าง ใน API ที่กำลังทำงานจริง

แอป Hono นี้เปิด collection ที่คุณสามารถอ่านและเพิ่มข้อมูลเข้าไปได้ เปิดมันใน StackBlitz แล้วลอง `GET /articles`, `GET /articles/1` และ `POST /articles`:

export const honoCode = `const articles = [{ id: 1, title: 'First post' }];

app.get('/articles', (c) => c.json(articles));

app.get('/articles/:id', (c) => {
  const id = Number(c.req.param('id'));
  const found = articles.find((a) => a.id === id);
  if (!found) return c.json({ error: 'Not found' }, 404);
  return c.json(found);
});

app.post('/articles', async (c) => {
  const input = await c.req.json();
  const created = { id: articles.length + 1, title: String(input.title ?? '') };
  articles.push(created);
  return c.json(created, 201, { Location: '/articles/' + created.id });
});`;

| Method | Safe? | Idempotent? | เหมาะกับ |
|---|---|---|---|
| GET | ✅ | ✅ | อ่านข้อมูล, search, filter |
| POST | ❌ | ❌ | สร้าง resource, trigger action |

## Common Mistakes

**ใช้ GET เพื่อ Trigger Side Effect**
อาการ:
- `GET /users/\{id\}/activate` — activate user ผ่าน GET
- bot, crawler, prefetch trigger action โดยไม่ตั้งใจ
- action ที่มี side effect ต้องใช้ POST, PUT, PATCH, หรือ DELETE

**ส่ง Sensitive Data ใน GET Query String**
อาการ:
- `GET /login?password=secret123`
- password ติด URL, server log, browser history
- ใช้ POST พร้อม body สำหรับ sensitive data

> 💡 Real World Example
>
> **GitHub API:**
> - `GET /repos/\{owner\}/\{repo\}` — อ่านข้อมูล repo (safe, idempotent)
> - `POST /repos/\{owner\}/\{repo\}/issues` — สร้าง issue ใหม่ (ไม่ idempotent)
>
> **Stripe API:**
> - `GET /charges/\{id\}` — ดู charge (cacheable)
> - `POST /charges` — สร้าง charge ใหม่ (ต้องการ idempotency key)

<Callout title="อย่าเปลี่ยนแปลงข้อมูลผ่าน GET">
ความผิดพลาดเรื่อง REST ที่พบบ่อยที่สุดอย่างหนึ่งคือการทำการกระทำผ่าน `GET` (เพราะมันง่ายที่จะ trigger จาก browser) ทั้ง cache, link prefetcher และ crawler ต่างสมมติว่า `GET` ปลอดภัยและอาจเรียกมันซ้ำ ๆ การเปลี่ยนแปลง state ใด ๆ ควรอยู่บน `POST`, `PUT`, `PATCH` หรือ `DELETE`
</Callout>
