ถ้า API ของคุณถูกเรียกจาก JavaScript บนเบราว์เซอร์ที่อยู่คนละ origin คุณจะได้เจอกับ CORS มันเป็นหนึ่งในส่วนที่ถูกเข้าใจผิดมากที่สุดของ web API ดังนั้นเรามาทำความเข้าใจให้ชัดเจนกันว่ามันคืออะไรและไม่ใช่อะไร

## CORS ทำหน้าที่อะไรกันแน่

same-origin policy ของเบราว์เซอร์จะบล็อกหน้าเว็บบน `https://app.example.com` ไม่ให้อ่าน response จาก `https://api.example.com` เว้นแต่ API จะอนุญาตอย่างชัดเจนด้วย header **Cross-Origin Resource Sharing** CORS ถูกบังคับใช้ **โดยเบราว์เซอร์** ในนามของ user — มันไม่ใช่การควบคุมการเข้าถึงที่ฝั่ง server

## simple request และ preflight

สำหรับ request แบบ "non-simple" (ส่วนใหญ่คือ JSON `POST`/`PUT`/`DELETE` และ header ที่กำหนดเอง) เบราว์เซอร์จะส่ง request **preflight** แบบ `OPTIONS` ไปก่อนเพื่อขออนุญาต:

export const preflightDiagram = `sequenceDiagram
  participant B as Browser
  participant S as API
  B->>S: OPTIONS /articles (preflight)
  S-->>B: 204 + Access-Control-Allow-Origin/Methods/Headers
  B->>S: POST /articles (actual request)
  S-->>B: 201 Created + Access-Control-Allow-Origin`;

header ใน response ที่สำคัญ: `Access-Control-Allow-Origin` (origin ใดบ้างที่อ่าน response ได้), `-Allow-Methods`, `-Allow-Headers` และ `-Allow-Credentials` (อนุญาตให้ส่ง cookie/credentials หรือไม่)

## การตั้งค่ามัน

export const honoCode = `const ALLOWED = new Set(['https://app.example.com']);

app.use('*', async (c, next) => {
  const origin = c.req.header('Origin');
  if (origin && ALLOWED.has(origin)) {
    c.header('Access-Control-Allow-Origin', origin);
    c.header('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE');
    c.header('Access-Control-Allow-Headers', 'Authorization, Content-Type');
  }
  if (c.req.method === 'OPTIONS') return c.body(null, 204); // preflight
  await next();
});

app.get('/articles', (c) => c.json([{ id: 1 }]));`;

จงหลีกเลี่ยงความผิดพลาดคลาสสิกของการสะท้อน origin ใดก็ได้ (`Access-Control-Allow-Origin: *`) **ควบคู่กับ** `Allow-Credentials: true` — การใช้ร่วมกันแบบนี้เป็นสิ่งต้องห้าม และที่สำคัญกว่านั้นคือมันจะเปิดเผย response ที่ผ่าน authenticated ให้กับเว็บไซต์ใดก็ได้

| CORS Header | ความหมาย | ตัวอย่าง |
|---|---|---|
| `Access-Control-Allow-Origin` | origin ที่อนุญาต | `https://app.example.com` หรือ `*` |
| `Access-Control-Allow-Methods` | HTTP method ที่อนุญาต | `GET, POST, PATCH, DELETE` |
| `Access-Control-Allow-Headers` | header ที่ client ส่งได้ | `Content-Type, Authorization` |
| `Access-Control-Max-Age` | cache preflight นานแค่ไหน | `86400` (1 วัน) |

## Common Mistakes

**`Access-Control-Allow-Origin: *` กับ Credentials**
อาการ:
- ตั้ง `Access-Control-Allow-Origin: *` แต่ต้องการส่ง cookie หรือ Authorization header
- browser block request — `*` ไม่ work กับ credentials
- ต้องระบุ origin จริง เช่น `https://app.example.com` เมื่อใช้ credentials

**Preflight ทุก Request เพราะ Cache ไม่ได้**
อาการ:
- ไม่ตั้ง `Access-Control-Max-Age` — browser ส่ง OPTIONS preflight ทุกครั้ง
- เพิ่ม latency ทุก request
- ตั้ง `Access-Control-Max-Age: 86400` เพื่อ cache preflight 1 วัน

> 💡 Real World Example
>
> **GitHub API:**
> - CORS ที่อนุญาตเฉพาะ origin ที่ register ผ่าน OAuth App
> - ไม่อนุญาต `*` สำหรับ authenticated endpoint
>
> **Stripe.js:**
> - browser SDK เรียก Stripe API โดยตรง — CORS config อนุญาตเฉพาะ Stripe domain
> - API Key ที่ใช้ใน browser เป็น publishable key เท่านั้น (ไม่มีสิทธิ์ charge)

<Callout title="CORS ไม่ใช่ authentication">
การผ่อนคลาย CORS ไม่ได้ทำให้ API ของคุณปลอดภัยน้อยลงต่อ client ที่ไม่ใช่เบราว์เซอร์ — curl, mobile app และ server ต่างเพิกเฉยต่อมันโดยสิ้นเชิง CORS เพียงตัดสินว่า web origin ใดบ้างที่เบราว์เซอร์จะยอมให้อ่าน response ของคุณ จงรักษา allowlist ของ origin ให้แคบไว้ และอย่าพึ่งพา CORS เป็นการควบคุมการเข้าถึงเด็ดขาด นั่นเป็นหน้าที่ของ authn/authz
</Callout>
