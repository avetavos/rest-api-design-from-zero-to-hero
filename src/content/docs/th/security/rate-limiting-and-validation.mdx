## Rate limiting

Rate limiting จำกัดจำนวน request ต่อ client ต่อช่วงเวลา (window) เพื่อป้องกันการใช้งานในทางที่ผิด, client ที่ทำงานผิดเพี้ยน และลูปที่เกิดขึ้นโดยไม่ตั้งใจ เมื่อผู้เรียกใช้เกินขีดจำกัด ให้ตอบกลับ `429 Too Many Requests` พร้อมกับ header `Retry-After` ที่บอกพวกเขาว่าควรลองใหม่เมื่อไหร่ ถือเป็นแนวปฏิบัติที่ดีที่จะประกาศ budget ด้วย header `RateLimit-Limit`, `RateLimit-Remaining` และ `RateLimit-Reset`

algorithm ที่พบบ่อยคือ **token bucket**: แต่ละ client มีถังหนึ่งใบที่เติม token กลับเข้าไปด้วยอัตราคงที่ แต่ละ request ใช้ token ไปหนึ่งตัว ถังที่ว่างเปล่าหมายถึง `429`

export const bucketCode = `function makeBucket(capacity, refillPerSec) {
  let tokens = capacity;
  let last = 0; // logical time in seconds
  return function take(now) {
    tokens = Math.min(capacity, tokens + (now - last) * refillPerSec);
    last = now;
    if (tokens >= 1) { tokens -= 1; return { ok: true, remaining: Math.floor(tokens) }; }
    return { ok: false, status: 429, retryAfter: Math.ceil((1 - tokens) / refillPerSec) };
  };
}

const take = makeBucket(3, 1); // burst of 3, refill 1/sec
console.log(take(0), take(0), take(0)); // three allowed
console.log(take(0));                    // 4th: 429 with retryAfter`;

## validate ทุกอย่าง

ทุกค่าที่มาจาก client เชื่อถือไม่ได้จนกว่าจะผ่านการ validate ความเสี่ยงหลัก ๆ ได้แก่:

- **Injection** — อย่าสร้าง SQL/command ด้วยการต่อ string เด็ดขาด ให้ใช้ parameterized query
- **Mass assignment** — อย่าเอา request body ไป spread ทับลงบน model แบบไม่ดูตาม้าตาเรือ ให้รับเฉพาะ field ที่อยู่ใน allowlist เท่านั้น มิฉะนั้น user อาจตั้งค่า `isAdmin: true` ได้
- **Oversized payloads** — จำกัดขนาด request body และความยาวของ array เพื่อหลีกเลี่ยงการทำให้ resource หมดเกลี้ยง
- **Type and range** — validate ชนิดข้อมูล, ความยาว และช่วงค่าด้วย schema และปฏิเสธด้วย `422` (ดู [Validation Errors](/rest-api/th/status-and-errors/validation-errors/))

| Benefits (Rate Limiting) | Trade-offs |
|---|---|
| ป้องกัน abuse และ DDoS | legitimate spike traffic อาจ fail |
| ป้องกัน credential stuffing และ brute force | ต้องออกแบบ limit ที่เหมาะสม — ไม่เข้มเกินหรือหย่อนเกิน |
| ลด cost ของ downstream service เช่น LLM API | rate limit shared ใน distributed system ต้องใช้ Redis |
| บอก client ผ่าน header ว่าเหลือ quota เท่าไร | user experience แย่ถ้า error ไม่อธิบาย |

## Common Mistakes

**Rate Limit โดยไม่บอก Client**
อาการ:
- 429 Too Many Requests โดยไม่มี header บอกว่าเหลือ quota เท่าไรและ reset เมื่อไร
- client ไม่รู้ว่าต้องรอนานแค่ไหน — retry loop ทำให้แย่ลง
- ส่ง `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` เสมอ

**Input Validation ที่ไม่ Sanitize**
อาการ:
- รับ `name: "<script>alert(1)</script>"` แล้วเก็บและแสดงโดยไม่ sanitize
- XSS เมื่อ API response ไปแสดงใน web app
- validate type/format และ sanitize string input ก่อน store

> 💡 Real World Example
>
> **GitHub API:**
> - rate limit: 5000 requests/hour สำหรับ authenticated request
> - header: `X-RateLimit-Limit: 5000`, `X-RateLimit-Remaining: 4999`, `X-RateLimit-Reset: 1609459200`
>
> **Stripe API:**
> - rate limit ตาม endpoint: `/charges` มี limit ต่างจาก `/events`
> - 429 พร้อม `Retry-After` header — client รู้ว่าต้องรอกี่วินาที

<Callout title="allowlist สำหรับ input, throttle สำหรับการใช้งานในทางที่ผิด">
รับเฉพาะ field และรูปแบบที่คุณคาดหวังไว้เท่านั้น — allowlist เหนือกว่า blocklist ทุกครั้ง และมันหยุดการโจมตีแบบ mass-assignment ได้อย่างเด็ดขาด จับคู่สิ่งนั้นเข้ากับ rate limit ต่อ client และการจำกัดขนาด body เพื่อให้ผู้เรียกใช้คนเดียวไม่สามารถทั้งทำลายข้อมูลของคุณหรือทำให้ขีดความสามารถของคุณหมดลงได้
</Callout>
