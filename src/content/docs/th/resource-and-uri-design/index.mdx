ใน REST นั้น **resource** คือ abstraction หลัก resource คือสิ่งใดก็ตามที่มีค่าพอจะตั้งชื่อและอ้างอิงถึง ไม่ว่าจะเป็น user, article, order หรือ shopping cart การออกแบบ API ส่วนใหญ่ก็คือการตัดสินใจว่า resource ของคุณคืออะไร และ URI ของพวกมันอ่านออกมาเป็นอย่างไร

## Resource และความสัมพันธ์ระหว่างกัน

แต่ละกล่องจะกลายเป็น collection ที่คุณสามารถอ้างถึงได้ (`/users`, `/articles`, `/comments`) และความสัมพันธ์ระหว่างกันก็เป็นใบ้ให้เห็นว่า URI จะ nest และ link กันอย่างไร

## โมดูลนี้ครอบคลุมอะไรบ้าง

- **Modeling resources** — หาคำนามที่เหมาะสมและระดับความละเอียด (granularity) ที่เหมาะสม
- **URI naming** — แนวปฏิบัติที่ทำให้ API คาดเดาได้
- **Nesting & relationships** — เลือกระหว่าง sub-resource กับการ link
- **Consistency & conventions** — รักษาให้ทั้ง API สอดคล้องกัน

<Callout title="Resource คือสิ่งของ ไม่ใช่กระบวนการ">
ถ้าคุณพบว่าตัวเองอยากได้ `/createUser` หรือ `/getArticleList` ให้ถอยกลับมาตั้งหลัก: resource คือคำนาม (`/users`, `/articles`) ส่วน HTTP method เป็นตัวจัดหา verb ให้ คำกริยาในเส้นทาง (path) คือสัญญาณที่ชัดเจนที่สุดว่า API ได้หลุดออกจาก REST ไปทาง RPC แล้ว
</Callout>
