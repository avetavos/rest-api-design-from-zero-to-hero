OpenAPI (เดิมเรียกว่า Swagger) คือวิธีมาตรฐานที่เครื่องอ่านได้ในการอธิบาย REST API ทั้ง path, operation, parameter, schema ของ request/response และ status code เอกสารเพียงไฟล์เดียวขับเคลื่อนทั้งเอกสาร การสร้าง client, mock server และการทดสอบ

## คำอธิบายแบบย่อที่สุด

นี่คือ endpoint หนึ่งที่อธิบายด้วย OpenAPI 3.1 (YAML)

```yaml
openapi: 3.1.0
info:
  title: Blog API
  version: 1.0.0
paths:
  /articles/{id}:
    get:
      summary: Get an article by id
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: string
      responses:
        '200':
          description: The article
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Article'
        '404':
          description: Not found
components:
  schemas:
    Article:
      type: object
      required: [id, title]
      properties:
        id: { type: string }
        title: { type: string }
        body: { type: string }
```

แต่ละ operation จะระบุ parameter ของมันและทุก response ที่มันสามารถคืนกลับได้ โดยแต่ละอย่างผูกกับ schema ส่วน `components/schemas` กำหนดรูปแบบที่นำกลับมาใช้ซ้ำได้ ซึ่งถูกอ้างอิงด้วย `$ref`

## spec-first กับ code-first

- **Spec-first** — เขียนเอกสาร OpenAPI ก่อน แล้วจึงสร้างจริงให้ตรงตามนั้น contract ถูกตกลงกันก่อนที่จะมี code เหมาะมากสำหรับทีมและผู้ใช้งานภายนอก
- **Code-first** — ใส่ annotation ให้ handler/schema ของคุณแล้วสร้าง spec ออกมาจากมัน ความคลาดเคลื่อนระหว่าง code กับ spec น้อยลง เพราะฝ่ายหนึ่งผลิตอีกฝ่ายหนึ่งออกมา

ไม่ว่าทางไหน เครื่องมือก็เปลี่ยน spec ให้เป็นคุณค่าได้ **Swagger UI** / Redoc แสดงเอกสารแบบโต้ตอบได้ ตัวสร้างผลิต client ที่มี type และตัว validator ตรวจสอบ request เทียบกับมัน

| Benefits (OpenAPI) | Trade-offs |
|---|---|
| documentation generate จาก schema — ไม่ drift จาก code | setup OpenAPI spec ต้องการ discipline |
| SDK generation อัตโนมัติ สำหรับหลายภาษา | spec ที่ verbose — YAML/JSON file ใหญ่ |
| validation middleware จาก schema — request/response ถูกต้องเสมอ | code-first vs spec-first ต้องเลือก approach |
| contract testing ระหว่าง producer และ consumer | spec เก่า ถ้า developer ไม่ update พร้อมกับ code |

## Common Mistakes

**OpenAPI Spec ที่ Drift จาก Implementation**
อาการ:
- เขียน spec ก่อน แล้ว implement ตาม — แต่ implementation เปลี่ยนโดยไม่ update spec
- developer ใช้ spec ที่ผิดสร้าง client — bug ตอน integrate
- generate spec จาก code (code-first) หรือ enforce spec validation ใน CI

**ไม่มี Example ใน Spec**
อาการ:
- spec มีแค่ type definition ไม่มี example request/response
- developer ไม่รู้ว่า field ควรส่งค่าอะไร — ต้องลองเอง
- เพิ่ม `example` หรือ `examples` field ใน spec ทุก request/response body

> 💡 Real World Example
>
> **Stripe API:**
> - OpenAPI spec ที่ comprehensive — ทุก endpoint มี schema, example, error response
> - ใช้ generate SDK สำหรับ 7+ ภาษา: Python, Ruby, Node.js, Go, Java, PHP, .NET
>
> **GitHub API:**
> - OpenAPI spec เปิดสาธารณะ — community สร้าง SDK และ tool บน spec นั้น
> - Octokit ทุกภาษา generate มาจาก spec เดียวกัน

<Callout title="ปล่อยให้ spec ทำงานซ้ำ ๆ แทน">
เอกสาร OpenAPI ที่ดูแลรักษาไว้ดีจะคุ้มค่าในตัวมันเอง ทั้งเอกสารแบบโต้ตอบ client ที่มี type ที่ถูกสร้างขึ้น mock server สำหรับทีม front-end และการ validate request ล้วนมาจากไฟล์เดียวกัน เลือก spec-first หรือ code-first ตาม workflow ของทีม แต่อย่าเขียนและดูแลคำอธิบายของ API เดียวกันด้วยมือถึงสามชุดแยกกัน
</Callout>
